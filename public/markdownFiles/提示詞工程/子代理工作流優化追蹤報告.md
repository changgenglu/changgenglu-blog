## 三、未來可能遇到的風險與挑戰

### 3.1 代理數量膨脹（Agent Sprawl）

**風險**：目前已有 16 個代理。隨著團隊持續優化，可能新增更多專項代理（如 `migration-reviewer`、`api-contract-validator`），導致：
- 主 agent 的調度邏輯越來越複雜
- 使用者難以理解「什麼情境該用什麼代理」
- 每個代理的 system prompt 載入成本累積

**緩解策略**：
- 設定代理數量軟上限（建議 ≤ 20）
- 新增代理前先評估「能否合併到現有代理的 skill 載入機制」
- 定期審計代理使用頻率，低頻代理（< 5% 使用率）考慮合併或移除

### 3.2 Context Window 壓力隨流程複雜度增長

**風險**：本次新增了多項規則（審查分級、收斂機制、自檢、迭代限制、Scope Creep 偵測），CLAUDE.md 已從 367 行增至 ~395 行。每次新增規則都增加主 agent 的 context 負擔，可能導致：
- 舊規則被 AI 「遺忘」（注意力稀釋）
- 矛盾規則之間的優先級不明確

**緩解策略**：
- 維持前述評估結論：500 行為拆分門檻
- 定期精簡——每 3 個月審視 CLAUDE.md，移除已固化為代理行為的規則
- 用 `.claude/rules/` 分離「被引用型規則」和「調度邏輯」

### 3.3 Wave 2 平行衝突在複雜場景中的殘餘風險

**風險**：即使新增了 Service Interface 契約，以下場景仍可能出問題：
- `@logic-implementer` 在實作中發現需要修改介面（新增參數、改變回傳型別）
- 契約定義在架構設計階段，此時可能對實作細節理解不夠深入
- 兩個代理修改同一個 Route 檔案（`foundation-implementer` 建立骨架，`api-implementer` 可能修改）

**緩解策略**：
- 在 `logic-implementer` 的「遇到規劃未涵蓋問題」步驟中，明確加入「修改 Service 介面必須暫停回報」
- 長期考慮：引入「契約鎖」機制——Wave 2 開始後，介面契約變更需要主 agent 協調

### 3.4 審查分級可能被 AI 利用來跳過審查

**風險**：主 agent 判斷「微型變更 → 免審查」時，可能低估變更的影響。例如：
- 看似只改一行的 `config` 修改，實際影響全局行為
- AI 可能（非惡意地）傾向將變更歸類為「微型」以減少後續步驟

**緩解策略**：
- 在審查分級表格中加入「安全相關變更一律走完整審查」的硬規則（已包含：「涉及安全/資料 → 中型以上」）
- 定期抽檢「免審查」的變更，評估是否有遺漏的問題

### 3.5 `/tmp/` 清理步驟的可靠性

**風險**：已加入「清理 `/tmp/*-latest.md` 殘留」步驟，但：
- 主 agent 可能在 context 過載時遺忘此步驟
- 若清理時誤刪了正在進行的任務的產出（如並行 session），會導致資料遺失

**緩解策略**：
- 長期方案：使用帶任務 ID 的子目錄（`/tmp/{task-id}/`）而非固定檔名
- 短期：觀察是否真的發生殘留問題，再決定是否加強

### 3.6 多人+ 多 Session 的協作風險

**風險**：當前設計假設「一個使用者、一個 session、一個任務」。但實際場景可能有：
- 同一使用者開兩個 Claude Code session 處理不同任務
- 團隊中多人使用同一專案的子代理配置

此時 `/tmp/` 的固定檔名、`project-decisions.md` 的追加寫入，都可能產生衝突。

**緩解策略**：
- 短期：在 CLAUDE.md 中加入「禁止平行處理兩個功能開發任務」的約束（已由 worktree 隔離部分緩解）
- 長期：引入 session ID 或 branch name 作為產出檔案的命名空間

### 3.7 代理定義更新的版本控制風險

**風險**：本次一口氣修改了 11 個檔案。若未來持續迭代，代理定義之間的**一致性**可能逐漸偏移：
- 新增代理時忘記更新其他代理的「後續代理」引用
- 修改某個代理的輸出格式但忘記更新下游代理的輸入解析
- 工作流程步驟號碼變更但回報點表格未同步

**緩解策略**：
- 每次修改代理定義後，執行「全局一致性檢查」（搜尋所有 `@agent-name` 引用，確認都存在）
- 考慮建立自動化 lint 腳本：掃描所有 agents/*.md 的「後續代理」區塊，驗證引用的代理名稱存在於代理清單中
- 將本次建立的驗證步驟（搜尋 `@code-reviewer`、`implementer（haiku）`）固化為可重用的檢查流程
