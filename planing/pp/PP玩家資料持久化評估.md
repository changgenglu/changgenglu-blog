# PP 玩家資料持久化評估

使用者身分驗證邏輯對比

  MG 平台（使用 Users 表）

  Users 表結構：
  @Entity('users')
  @Unique(['groupId', 'username'])
  export class Users {
      id: number;              // 主鍵
      groupId: number;         // 營運商分組
      username: string;        // 玩家帳號（唯一性：groupId + username）
      player: string;          // UUID，遊戲供應商的玩家 ID
      createdAt: Date;         // 建檔時間
  }

  認證流程：
  1. 星城系統透過 HTTP + JWT 傳來 playerId（既 MG 的 player UUID）
  2. getPlayerInfo() 從 Redis 快取查詢玩家狀態（xinkey、isLogin 等）
  3. 若 Redis 缺失，Users 表用於記錄帳號與 UUID 的對應
  4. 玩家實時資訊存於 Redis：
    - Key：puppy:player:{groupId}:{playerId}
    - Value：{ username, xinkey, vipLevel, isLogin }
    - TTL：無期限（只在登出時刪除）

  PP 平台（無 Users 表）

  認證流程：
  1. PP 呼叫 /authenticate.html 並傳遞 token
  2. Puppy 透過 Token 查詢 Redis：puppy:pp_token:{token}
  3. Redis 中儲存：{ userId, xinkey, currency, createdAt }（TTL：30 分鐘）
  4. 玩家資訊快取：
    - Key：puppy:pp_player:{groupId}:{userId}
    - Value：{ xinkey, isLogin }
    - TTL：190 分鐘

  關鍵區別： 不需要持久化的使用者帳號表，因為玩家身份透過 PP 發行的 Token 完全驗證

  ---
  PP 是否需要 Users 表？

  評估結論：不需要

  原因分析

  ┌──────────────┬───────────────────────────────┬───────────────────────────────────┐
  │     維度     │       MG（有 Users 表）       │         PP（無 Users 表）         │
  ├──────────────┼───────────────────────────────┼───────────────────────────────────┤
  │ 玩家標識來源 │ 星城系統發行 playerId（UUID） │ PP 發行 token，userId 在 token 中 │
  ├──────────────┼───────────────────────────────┼───────────────────────────────────┤
  │ 帳號對應     │ 需要 username → playerId 對應 │ Token 本身就是身份憑證            │
  ├──────────────┼───────────────────────────────┼───────────────────────────────────┤
  │ 持久化需求   │ 是（多次登錄共用帳號）        │ 否（Token 一次性，30 分鐘有效）   │
  ├──────────────┼───────────────────────────────┼───────────────────────────────────┤
  │ 跨客戶端支援 │ 需要（多裝置同帳號）          │ 不需要（Token 驗證單一客戶端）    │
  ├──────────────┼───────────────────────────────┼───────────────────────────────────┤
  │ 帳號管理     │ 複雜（帳號狀態、VIP 等級）    │ 簡單（只需驗證 Token 有效性）     │
  └──────────────┴───────────────────────────────┴───────────────────────────────────┘

  現有架構已完全滿足需求

  1. Token 驗證 — PP Token 存於 Redis（30 分鐘 TTL），完全足夠一場遊戲週期
  2. 玩家狀態快取 — getPlayerInfo() 儲存 xinkey 與 isLogin，190 分鐘 TTL
  3. 交易隔離 — PpBetRecord 與 PpBetRecordSub 已記錄完整交易歷史
  4. 冪等保障 — Redis 交易快取（190 分鐘）確保重複請求返回相同結果

  若要新增 Users 表，代價與風險

  - 代價：
    - 增加資料庫設計（新 Entity）
    - 新增資料庫遷移
    - 修改 PP 模組初始化邏輯
    - 額外的查詢成本
  - 風險：
    - 同步問題：Redis Token 過期 vs DB 記錄不一致
    - 幻讀：Token 已失效但 Users 表仍有記錄
    - 跨地區部署複雜度增加

  ---
  建議

  現有架構足夠。不建議新增 Users 表。

  理由：
  1. PP 完全基於 Token，Token 本身就是玩家身份驗證的全部根據
  2. Redis 快取層 已涵蓋所有臨時狀態需求
  3. 交易記錄 以 roundId + reference 為主鍵，無需帳號層級的持久化

  若未來需求變更（如多裝置跨設備支援），再評估是否需要帳號表。
