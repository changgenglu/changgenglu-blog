# Pragmatic Play 無縫錢包 API 整合核心規範 (V4 完美對齊終極版)

_基於 API 版本 3.222，融合資安防護、財務防呆、頻率限制與系統架構設計要求。_

## 1. 系統核心安全與機制 🛡️

### 1.1 嚴格的哈希驗證 (Hash Calculation) 與簽名防護

為避免簽名驗證失敗或潛在的簽名繞過漏洞，計算 MD5 Hash 時必須嚴格遵守以下順序與規則：

1. **空值排除**：從 POST 請求中獲取所有參數，**如果參數的值為空 (empty/null)，則絕對不可加入串接字串中**。
2. **字典排序**：將剩餘的參數名稱按字母順序排列。
3. **字串串接**：以 `key1=value1&key2=value2` 的格式串接（注意：需確保特殊字元已正確轉義/Url Encoding，以免破壞字串結構）。
4. **附加密鑰與雜湊**：在字串最尾端直接附加上您的 `SECRET`，最後使用 MD5 計算。若失敗，必須回傳錯誤碼 `5`。

### 1.2 玩家生命週期與幣別鎖定邏輯 (Lifecycle & Currency Lock)

- **唯一入口**：`Authenticate` 是 PP 系統建立與驗證玩家的唯一入口。若玩家不存在，會自動建立帳戶。
- **幣別不可變性 (⚠️ 核心鎖定)**：玩家的幣別 (`currency`) 會在首次開啟遊戲並建立帳戶時**永遠鎖定**。創建玩家帳戶後，絕對無法更改貨幣。我方系統在每次驗證時，務必檢查本地幣別與傳入 PP 的幣別是否一致。

### 1.3 核心識別碼與【金額格式定義】

- **金額格式 (Decimal 10,2)**：PP 系統全線的資金欄位（如 `amount`, `cash`, `win` 等）皆為**帶有兩位小數的 Decimal 格式（例如：`100.00`）**。切勿使用純整數 (int) 解析，以免產生 0.01 單位的帳務偏差。
- **`reference`**：PP 系統產生的唯一交易參考號（String 32）。
- **`transactionId`**：運營商系統產生的唯一交易 ID。建議直接將接收到的 `reference` 作為此值回傳。

### 1.4 A04:2021 安全性防護：非同步派彩防護與唯一複合索引

針對非同步派彩（如 `BonusWin`, `PromoWin`, `JackpotWin`），惡意攻擊者或不穩定的網路環境可能導致離線派彩請求被重複發送。

- **冪等性與唯一索引 (Idempotency & Unique Index)**：API 文件明確規定所有交易呼叫皆為「冪等元 (idempotent)」。我方資料庫**必須**針對 `reference` 建立「唯一複合索引」。
- **防禦機制**：收到重發請求時，利用 Unique Index 阻擋重複寫入。若發現已處理過，只需回傳玩家「當下實際最新餘額」與原本的 `transactionId` 即可，**絕不可重複加/扣款**。

## 2. API 頻率限制與流量控制 (Rate Limits) 🚦

為避免伺服器遭 PP 阻擋 (回傳 Error 99) 或引發效能瓶頸，主動呼叫 PP API 時必須嚴格遵守以下官方頻率限制：

- **健康檢查 (HealthCheck)**：單一品牌調用頻率**不超過每分鐘 1 次**。
- **重播連結 (Replay link)**：單一娛樂場**每秒最多發送 1 個請求**。
- **資料傳送與報表 (Data Feeds)**：任何資料傳送 API 請求**不應超過每分鐘 1 次**。
- **失敗轉帳重試**：每分鐘 1 次，總重試時間不得長於 1 小時。
- **錦標賽相關 API**：
  - 贏家名單 (Winners)：每 60 分鐘最多 1 次。
  - 活躍賽事 (Active) / 獎品配置 (Prizes)：每 30 分鐘最多 1 次。
  - 排行榜 (Leaderboard) / 天降獎勵 (Prize Drop)：每 10 分鐘最多 1 次。

## 3. 遊戲啟動與身分驗證

### 3.1 取得遊戲連結 (`POST /game/url` - 運營商呼叫 PP)

亞洲地區運營商**強制**使用此 API。

- **必填參數**：`secureLogin`, `symbol`, `language`, `token` (單次安全令牌), `externalPlayerId`, `country`, `hash`。
- **促銷權限 (`extraInfo`)**：強烈建議在 `Authenticate` 響應中透過 `extraInfo` 的 `promoAvailable` 來控制玩家是否能參與促銷活動，優先級高於啟動參數。

### 3.2 玩家身分驗證 (`POST /authenticate.html` - PP 反向呼叫運營商)

- **必填參數**：`hash`, `token`, `providerId`。
- **必填響應**：`userId`, `currency`, `cash`, `bonus`, `error`, `description`。

## 4. 常規交易 API (PP 呼叫運營商)

_運營商需在伺服器實作這些 HTTP POST 端點，並嚴格依賴 `reference` 處理冪等性。_

### 4.1 下注扣款 (`POST /bet.html`)

玩家點擊旋轉時觸發。若餘額不足需回傳錯誤代碼 `1`。

- **財務對帳關鍵 `usedPromo`**：若該次下注有部分扣除玩家的獎勵餘額 (bonus)，必須在響應中回傳實際扣除數值；若全為真錢 (cash)，則回傳 `"usedPromo": 0`。

### 4.2 贏分派彩 (`POST /result.html`)

將贏得的 `amount` 加回玩家現金餘額中。

- **促銷獎金攔截 (天降獎勵 PromoWin)**：當有營銷活動時，此請求可能會額外帶有 `promoWinAmount`。**我方系統必須將 `promoWinAmount` 一併加回玩家的 cash 餘額中**。

### 4.3 🌟 真人遊戲餘額調整 (`POST /adjustment.html` - 實作補強)

這是 **Live Casino (真人娛樂場)** 不可或缺的核心 API，支援離線發送。

- **機制說明**：處理真人遊戲中的退款或特殊派彩等額度調整。
- **正負值處理**：`amount` 可能為正值（加錢）或**負值（扣錢）**。
- **防呆規範**：當 `amount` 為負值且玩家餘額不足時，運營商**必須回傳錯誤代碼 `1` (Insufficient balance)** 拒絕扣款。
- **必填參數**：需記錄隨同發送的 `validBetAmount` (有效投注額)，供洗碼量/報表使用。

### 4.4 交易退款 (`POST /refund.html`)

用於取消下注。若運營商系統找不到對應下注單號，不需做資金操作，直接回傳成功 (`0`) 即可。

## 5. 特殊玩法結算 (非同步/背景發送)

_⚠️ 此類 API 極易遭受重放攻擊，極可能在玩家離線時發送，請依賴第 1.4 節的防護機制。_

### 5.1 免費旋轉結算 (`POST /bonusWin.html`)

此呼叫是異步的且與回合結束無關。所有免費次數轉完後，PP 會一次性發送總贏分。

### 5.2 累積大獎派彩 (`POST /jackpotWin.html`) 與獎金拆分

- **財務對帳陷阱**：PP 官方僅向運營商支付「累積獎金 (progressive)」。
- **拆分實作**：請務必聯絡 PP 開啟選填的 `jackpotDetails` 參數。回傳格式為 JSON 字串 (如：`"{\"progressive\":130.34,\"non-progressive\":3125.0}"`)，我方需解析後分別入帳。

### 5.3 促銷與錦標賽獨立派彩 (`POST /promoWin.html`)

錦標賽結束後延遲約 30 分鐘於背景發送，獎金直接加回現金餘額。

---

## 6. 複雜資料與資料庫溢位防呆

### 6.1 `roundDetails` 格式說明與欄位長度限制 ⚠️

- **風險描述**：PP 傳送的 `roundDetails` 資料類型定義為 **`String(4000)`**。
- **防呆要求**：我方資料庫與後端接收緩衝區**必須將此欄位長度設定為 `VARCHAR(4000)` 或 `TEXT` 以上**，否則極易引發字串截斷或系統崩潰。
- **解析範例**：
  - **老虎機**：`"spin,totalBet:200.0,freeSpinCount:18,totalWin:284.0..."`。包含功能標記如 `spin`, `bonusBuy`, `anteBet`, `superSpin`。
  - **真人遊戲**：`"type:bets,desc:[{\"c\":\"Even\",\"a\":\"3.0\"},{\"c\":\"Red\",\"a\":\"3.0\"}]"` (JSON 陣列字串)。

## 7. 選用機制與進階規範

### 7.1 回合結束通知 (`POST /endRound.html`)

官方強烈建議：若不需要即時結束交易，請在 PP 側**禁用**此功能以減少負載。若法規要求獲取詳細開獎結果 (`POST /roundDetails.html`)，則必須啟用此功能。

### 7.2 未完成回合自動完成 (Auto-finalization)

針對中途斷線的玩家，系統在設定逾時後自動於背景發送退款 (`refund`) 或結算派彩 (`result`)。

- 若玩家已轉出並贏得額外免費旋轉，將視為有效旋轉，**資金不予退還**。
- 預設為關閉，需主動聯絡 PP 技術支持申請配置。

## 8. 完整核心錯誤代碼 (Error Codes 總覽)

運營商在響應 PP 系統發送的無縫錢包 API 呼叫時，應依據以下代碼回傳錯誤狀態。請特別注意內部異常的重試行為差異。

|  代碼   | 說明 (Description)                       | 觸發情境與系統協調重試 (Reconciliation) 邏輯                                                                                          |
| :-----: | ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
|  **0**  | **Success (成功)**                       | 交易順利完成，無須重試。                                                                                                              |
|  **1**  | **Insufficient balance (餘額不足)**      | 下注 (`bet`) 或真人調整 (`adjustment` 負值) 時餘額不足。PP 將**發送退款 (Refund)** 或排程重試。                                       |
|  **2**  | **玩家未找到或已登出**                   | PP 發送任何請求時，若運營商端找不到玩家或玩家已登出，則回傳。PP 會重試。                                                              |
|  **3**  | **不允許下注**                           | 玩家因特殊獎金或限制，不被允許遊玩特定遊戲時回傳。PP 將排程重試。                                                                     |
|  **4**  | **令牌無效或過期**                       | `authenticate` 階段玩家身分驗證失敗 (Token 無效/未找到/過期) 時回傳。PP 會重試。                                                      |
|  **5**  | **Invalid hash code (Hash 無效)**        | 任一請求 Hash 驗證失敗時回傳。PP 將排程重試。                                                                                         |
|  **6**  | **玩家被凍結**                           | 若玩家帳號被封禁或凍結，於任何請求回傳此錯誤。PP 會重試。                                                                             |
|  **7**  | **請求參數錯誤**                         | 請求參數錯誤或必填參數遺失，請檢查 POST 參數。PP 會重試。                                                                             |
|  **8**  | **遊戲未找到或已禁用**                   | 遊戲無法進行時於下注請求 (`bet`) 回傳。⚠️ **注意**：即使遊戲被禁用，帶有贏獎金額的派彩請求 (`result`) 仍應正常接收並處理。PP 會重試。 |
| **50**  | **已達到投注限額**                       | 針對受監管市場。當下注達到上限時回傳。PP 將排程重試。                                                                                 |
| **99**  | **已超出請求限制**                       | 運營商主動呼叫 PP 時觸發 API 頻率限制 (Rate Limit) 時回傳。請稍後重試。                                                               |
| **100** | **Internal server error (需重試)**       | 運營商系統內部異常且「需要」重試。**請求將進入 PP 的協調流程 (Reconciliation)** 不斷重發。                                            |
| **120** | **Internal server error (不重試)**       | 運營商系統內部異常但邏輯上「不需要」重試。**請求將不會進入協調流程**。                                                                |
| **130** | **EndRound 內部伺服器錯誤**              | 僅限 `EndRound` 方法專用。系統無法處理且需要重試時回傳。                                                                              |
| **210** | **Reality check warning (現實檢查警告)** | 受法規要求，玩家遊戲時間達到設定的間隔時回傳，遊戲端將向玩家跳出警告彈窗。PP 會重試。                                                 |
| **310** | **超出投注限制 (動態變更)**              | 玩家下注金額超出運營商動態設定的限額時回傳 (與 `authenticate` 回傳的 limits 相關)。客戶端將提示玩家並要求重新開啟遊戲套用新限額。     |
