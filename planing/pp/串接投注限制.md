# 串接投注限制

  總部錢包 TCP Login 回應已經攜帶 max 和 min

  LoginMessage（stars_protocol/message/login_message.ts:10-12）：
  public balance: number;
  public max: number;    // 投注上限
  public min: number;    // 投注下限

  封包解析（stars_protocol/packet/response/login.ts:16-18）：
  message.balance = reader.getLong();
  message.max = reader.getUint32(true);
  message.min = reader.getUint32(true);

  但兩個平台的 executeLogin() 都丟棄了 max/min

  MG（mg.service.ts:277-279）：
  async executeLogin(xinkey: string): Promise<{ username: string; err: string }>
  // ← 只回傳 username 和 err，丟棄 max/min

  PP（pp.service.ts:251）：
  async executeLogin(xinkey: string): Promise<{ username: string; err: string }>
  // ← 同樣丟棄 max/min

  PP authenticate 回傳（pp.controller.ts:123-129）：
  return {
      userId: playerToken.userId,
      currency: playerToken.currency,
      cash: balanceResult.balance,
      bonus: 0,
      error: PpErrorCode.SUCCESS,
      // ← 缺少 betLimits
  };

  重新評估

  不需要在 Puppy 新增級距對應表。 原因：

  1. 星城 /api_login 回傳的 max/min 已經是該玩家等級對應的投注上下限，這是星城系統根據玩家 VIP
  等級計算好的結果
  2. TCP Login 回應已經解析了 max/min，資料已經在 LoginMessage 物件中
  3. Puppy 只需要透傳這兩個值，不需要自行維護級距對應表

  實際需要修改的是

  將 executeLogin 回傳的 max/min 串接到 PP authenticate 的回傳中：

  總部錢包 TCP Login → LoginMessage { max, min }
      → PP executeLogin() 回傳 max/min（目前丟棄）
          → PP authenticate 回傳中加入 betLimits（目前缺失）

  具體修改點：

  1. pp.service.ts:251 — executeLogin 回傳型別加入 max、min
  2. pp.controller.ts:96-129 — authenticate 端點將 max/min 轉換為 PP 的 betLimits 格式回傳
  3. dto.ts:382-391 — PpWalletResponse 加入 betLimits 欄位

  需要注意的是，PP betLimits 的 JSON 格式需要對照 PP
  串接文件的規範來構造（可能需要依遊戲類型區分）。星城回傳的是單一 max/min，但 PP
  可能要求按遊戲類型分別設定——這部分需要確認 PP 文件中 betLimits 的具體結構。
